Overview

stargzr runs as a single Rust binary pointed at a folder of audio and video files. it gives every browser on your network a media player. any user can start broadcasting — sharing their playback state in real time over a WebSocket — and others can tune in and hear exactly what the broadcaster is hearing, synchronized.

this document is structured as an execution walkthrough — it follows what actually happens in order, from the moment the binary starts to a listener hearing a seek. every section has an immediately readable summary, and expandable deep-dives that go into the exact code, the reasoning, and the edge cases.

01 — server boot

Starting Up

the binary's entry point is main.rs. it reads environment variables (only set under Docker), falls back to a media/ folder in the project root, then hands a PathBuf to initialize() in mod.rs. PathBuf is Rust's typed path: it knows how to join components, add extensions, and behaves correctly on both Unix and Windows.

environment and path

MEDIA_PATH is only set when running through Docker. in development the folder is assumed to be media/ at project root. PathBuf is Rust's heap-allocated path type — the compiler knows it represents a filesystem path, so it gives you methods like .join("subfolder"), .with_extension("mp4"), and OS-correct separator handling, rather than you doing string concatenation and hoping the slashes are right.

initialize() — order matters

once inside initialize() in mod.rs, things happen in this order because later steps depend on earlier ones:

  1. call init_logging() — must be first so subsequent log calls have a subscriber
  2. call metrics::init_metrics() — installs the Prometheus recorder globally before any counters fire
  3. bind the TCP listener — registers with the kernel, claims port 8083
  4. call init_player_state() — builds the playlist and AppState
  5. call create_player_router(state) — builds the Axum router with state attached
  6. spawn cleanup_stale_sessions as a background Tokio task
  7. call axum::serve() — starts the accept loop

binding the TCP listener

let listener = tokio::net::TcpListener::bind("0.0.0.0:8083").await
    .expect("Failed to bind to port 8083");

this registers a socket with the OS kernel and tells it to accept connection attempts on port 8083 from any IP. the kernel now queues incoming SYN packets. the application hasn't accepted any connections yet — it has just reserved the port and created the listening socket.

logging.rsrust

initializes the tracing subscriber. reads the ENVIRONMENT env variable and picks a log level and format accordingly.

what tracing is

Rust's tracing crate is structured logging. instead of println!, you call macros like tracing::info!(), tracing::debug!(), tracing::warn!(). these go to a subscriber — an object that decides what to do with each event. without a subscriber installed, all log calls are silently discarded. this is why init_logging() must run before anything else.

metrics.rs — OnceLock and the global recorder

static HANDLE: OnceLock<PrometheusHandle> = OnceLock::new();

pub fn init_metrics() {
    let handle = PrometheusBuilder::new()
        .install_recorder()
        .expect("Failed to install Prometheus recorder");
    HANDLE.set(handle).ok();
}

OnceLock is a cell that can only be written once and is then permanently readable. .set().ok() silently ignores a second call — if somehow init_metrics was called twice the second call is a no-op rather than a panic. the metrics crate needs a global recorder installed before any counter or gauge is incremented, so this must run before the router starts handling requests.

mod.rs — init_player_state()rust

scans the media folder, builds the initial playlist, creates the broadcast channel, constructs AppState, and wraps it in Arc.

async directory scan

if let Ok(mut entries) = tokio::fs::read_dir(&media_folder).await {

tokio::fs::read_dir is asynchronous — the I/O and reading does not block the runtime. each .await yields to Tokio so other tasks can run while the filesystem responds. the chained if let pattern means any failure (unreadable entry, unsupported file type, no metadata) silently skips that file without crashing.

media_type_for() lives in types.rs and is the single source of truth for what extensions are playable. it returns Some(MediaType::Audio) or Some(MediaType::Video) for supported formats, None for everything else. PDFs, ZIPs, and other files get None and are skipped from the playlist — they appear in the "Other" tab instead.

the broadcast channel

let (global_broadcast_tx, _) = broadcast::channel(100);

a Tokio broadcast channel is one-sender, many-receivers. the _ discards the initial receiver because no one is connected yet. later, each WebSocket connection subscribes via global_broadcast_tx.subscribe(). the capacity of 100 means up to 100 messages can be buffered for slow receivers before they start falling behind. for system-wide announcements that fire at most a few times per second, 100 is generous.

why Arc<AppState>

AppState is large — it contains multiple DashMaps, the playlist, channels, atomics, and a semaphore. if every handler function received a copy of it, every request would allocate and copy megabytes of data. Arc (atomic reference count) wraps it in a single heap allocation. every handler receives a clone of the Arc, which is just an atomic pointer increment — no data copying at all. when the last Arc clone is dropped, the data is deallocated. in practice the server runs until shutdown so that never happens.

mod.rs — create_player_router()rust

builds the Axum router, maps every URL to a handler function, mounts static file serving, and nests everything under the /stargzr prefix.

Axum is not a filesystem

many people assume a web server behaves like a filesystem browser — drop a file into a directory and it becomes accessible as a URL. traditional servers like Apache and Nginx reinforce this assumption because they expose directory structures by default. this model does not apply here.

Axum is a request router. a file existing on disk has zero visibility to the HTTP layer unless explicitly connected through routing configuration. disk state and HTTP state are independent systems. when the browser receives the HTML and encounters <link rel="stylesheet" href="/stargzr/static/css/player.css">, it makes a completely new HTTP request. without a matching route, that request returns 404 regardless of whether the file exists on disk.

ServeDir, from tower-http, connects HTTP paths to filesystem locations — it maps the remaining path segment onto a filesystem lookup. env!("CARGO_MANIFEST_DIR") is a compile-time constant that expands to the absolute path of the directory containing Cargo.toml. this is evaluated at compile time, not at runtime, so the path is baked into the binary.

the /stargzr prefix

Router::new().nest("/stargzr", inner)

the inner router is nested under /stargzr so every route has that prefix. this makes it easy to run the server behind a reverse proxy at a subpath. all JavaScript fetch calls and WebSocket URLs are built relative to window.location, so they automatically pick up the correct prefix.

defer scripts and load order

every <script> tag has the defer attribute. this means the browser downloads all scripts in parallel but does not execute any of them until the HTML is fully parsed. crucially, defer preserves document order — scripts execute in the order they appear in the HTML even if some finish downloading earlier. this is why the load order in the comments at the bottom of player.html is not just a suggestion: utils.js calls at parse time from inside other scripts, and main.js instantiates classes defined in the files above it.

mod.rs — NoDelayListener + ConnectInforust

wraps the TCP listener to set TCP_NODELAY on every accepted socket, and threads the peer address into request extensions so handlers can identify clients without touching raw sockets.

the problem: Axum hides the socket

normally you never touch individual TCP connections in Axum. the socket moves directly from the OS kernel into Axum's internal machinery. Hyper reads bytes, parses HTTP, routes to a handler, handler returns a response, Hyper writes bytes. the application sees only high-level Request and Response objects.

but certain TCP options must be applied after a connection is accepted but before it is used. Nagle's algorithm is one of these. and the peer's IP address is another — it is available at accept time but not from the request itself (which only has headers, and headers can be spoofed).

NoDelayListener — a thin wrapper

struct NoDelayListener(tokio::net::TcpListener);

impl axum::serve::Listener for NoDelayListener {
    type Io   = tokio::net::TcpStream;
    type Addr = std::net::SocketAddr;

    async fn accept(&mut self) -> (Self::Io, Self::Addr) {
        loop {
            match self.0.accept().await {
                Ok((stream, addr)) => {
                    stream.set_nodelay(true).ok();   // ← our window
                    return (stream, addr);
                }
                Err(e) => tracing::error!("Accept error: {}", e),
            }
        }
    }
}

Axum doesn't require a TcpListener specifically — it accepts anything implementing its Listener trait. by wrapping in a newtype and implementing that trait, we get a hook into the accept loop. set_nodelay(true) disables Nagle's algorithm for this connection. Nagle batches small outgoing packets to improve bulk throughput. WebSocket sync frames are tiny JSON strings that must arrive as fast as possible — batching adds latency on the critical seek path.

ConnectInfo and the orphan rule

#[derive(Clone)]
pub struct PeerAddr(pub std::net::SocketAddr);

impl axum::extract::connect_info::Connected<axum::serve::IncomingStream<'_, NoDelayListener>>
    for PeerAddr
{
    fn connect_info(target: axum::serve::IncomingStream<'_, NoDelayListener>) -> Self {
        PeerAddr(*target.remote_addr())
    }
}

the orphan rule: Rust only allows implementing a trait for a type if either the trait or the type is defined in your crate. Connected is from Axum and SocketAddr is from the standard library — both are foreign. implementing Connected for SocketAddr is illegal. the PeerAddr newtype is defined in this crate, so it satisfies the rule.

calling router.into_make_service_with_connect_info::<PeerAddr>() tells Axum: for every new TCP connection, run the Connected impl on the IncomingStream and store the result in the request's extensions. handlers extract it with ConnectInfo(addr): ConnectInfo<PeerAddr>. this happens once per TCP connection — all requests on the same connection share the same ConnectInfo. for WebSocket connections that is one extraction for the entire session's lifetime.

why not just read X-Real-IP everywhere?
behind a reverse proxy like Caddy, ConnectInfo always gives 127.0.0.1 because the proxy is the actual peer. so the WebSocket handler does prefer x-real-ip from the header. but the header can be spoofed by a client if there's no proxy — ConnectInfo cannot. both are used: header for the proxied case, ConnectInfo as the unauthenticated fallback.

AppState — the full reference

AppState is used everywhere. every handler and every WebSocket task receives an Arc<AppState>. understanding the design choices on each field explains why the whole system is lock-contention-free at scale.

playlist — RwLock<Vec>, not DashMap

pub playlist: Arc<RwLock<Vec<MediaInfo>>>

the playlist is a sequential structure accessed by integer index. DashMap is key-value; it wouldn't help here. RwLock allows many simultaneous readers (streaming, TuneIn, playlist fetches) while the write is exclusive (upload insert). a Mutex would have been wrong — it allows only one reader at a time even when no writing is happening, which would serialize every streaming range request.

it's wrapped in its own Arc separately from the outer Arc<AppState> so the lock scope is narrow and explicit. you never want to hold a lock across an .await point — a write lock held across an await is a deadlock waiting to happen. the inner Arc makes the borrow explicit and scoped.

DashMap everywhere else

a single RwLock<HashMap> serialises every write across the entire map. a heartbeat from broadcaster A would block a TuneIn from listener B even though they touch different keys. DashMap shards internally — by default into roughly 4× the number of CPU cores — so operations on different keys almost never contend.

two broadcast channels, two audiences

pub global_broadcast_tx: broadcast::Sender<Arc<PreparedMessage>>,
pub broadcast_channels:  DashMap<String, broadcast::Sender<Arc<PreparedMessage>>>

global_broadcast_tx reaches every connected WebSocket client. BroadcasterOnline, BroadcasterOffline, Analytics, ServerShutdown all go here. every client subscribes to it on connection.

the per-broadcaster channels in broadcast_channels reach only the listeners tuned to a specific broadcaster. Sync messages go here — a listener tuned to broadcaster A must never receive Sync from broadcaster B. the send task dynamically swaps which channel it's subscribed to when tuned_rx changes.

both carry Arc<PreparedMessage>. the message is serialized to JSON once at the send site, wrapped in Arc, and distributed. each receiver gets a pointer clone — a single atomic increment — not a copy of the string. with 50 listeners, one serialization and 50 pointer increments instead of 50 serializations.

atomics — lock-free counters

pub active_connections: AtomicUsize,
pub last_analytics_ms:  AtomicU64

AtomicUsize and AtomicU64 allow lock-free reads and writes using CPU-level atomic instructions. active_connections is incremented on connect and decremented on disconnect; it's read by broadcast_analytics to skip sending if zero clients are connected. last_analytics_ms uses compare_exchange to prevent multiple concurrent tasks from all sending analytics at once — only the one that successfully swaps the timestamp actually sends.

conversion_semaphore — one ffmpeg at a time

pub conversion_semaphore: Arc<Semaphore>  // initialized with count = 1

ffmpeg is CPU-intensive. running multiple conversions simultaneously saturates the CPU and makes all of them slower while starving other tasks. the semaphore with count 1 makes concurrent conversions queue rather than pile up. each upload that needs conversion calls acquire().await which either proceeds immediately if the slot is free, or suspends until the previous conversion releases it.

02 — first request

The First HTTP Request

a browser navigates to /stargzr/. Hyper reads bytes from the socket, parses the HTTP request, and Axum routes it to player_page in handlers.rs. Axum's extractor system runs before the function body — State(state) pulls the SharedState from request extensions, HeaderMap gives raw headers.

Axum extractors

extractors are types that implement the FromRequest or FromRequestParts trait. Axum resolves them before calling the handler function. State(state) pulls the attached SharedState from extensions — this is the same state that was attached with .with_state() when building the router. HeaderMap gives a reference to all HTTP headers. both are resolved transparently.

reading the playlist without holding the lock long

let (current_media, total_medias, is_video) = {
    let playlist = state.playlist.read().await;  // acquire read lock
    let media_info = playlist.get(current_index);
    let media = media_info.map(|s| s.filename.clone())
        .unwrap_or_else(|| "No medias found".to_string());
    let is_video = media_info
        .map(|s| matches!(s.media_type, MediaType::Video))
        .unwrap_or(false);
    (media, playlist.len(), is_video)
    // ← lock guard dropped here, lock released
};

the playlist lock is held only for the duration of the block. everything is cloned out. the guard drops at the closing brace, releasing the lock before any further async work happens. this is deliberate — holding a lock across an .await point would block every other task trying to read the playlist for the entire duration of whatever async operation follows.

is_video flag

whether the current track is video or audio is determined server-side and baked into the HTML template. this tells the browser which element to show initially — <audio> or <video> — before any JavaScript runs. without this, the page would flash the wrong element for a frame, or worse, try to play a video file through the audio element.

session.rsrust

cookie-based session tracking. no login, no passwords — just a UUID in a cookie. handles creation, position tracking, and cleanup.

how sessions are created

pub fn get_session_id(headers: &HeaderMap) -> String {
    headers.get(header::COOKIE)
        .and_then(|v| v.to_str().ok())
        .and_then(|cookies| {
            cookies.split(';').find_map(|c| {
                c.trim().strip_prefix("player_session=")
            })
        })
        .map(|s| s.to_string())
        .unwrap_or_else(|| Uuid::new_v4().to_string())
}

there is no login, no password, no database lookup. the cookie is a random UUID. if absent a new UUID is generated. stargzr is designed for a trusted home network where this is acceptable. the cookie is set with HttpOnly (JavaScript can't read or modify it), SameSite=Strict (only sent on same-origin requests, preventing CSRF), and Max-Age=86400 (24 hours).

get_or_create_position — fast and slow paths

pub fn get_or_create_position(state: &AppState, session_id: &str) -> usize {
    // fast path — cheap read, no exclusive lock
    if let Some(session) = state.sessions.get(session_id) {
        return session.current_index;
    }
    // slow path — create entry atomically
    state.sessions
        .entry(session_id.to_string())
        .or_insert_with(|| {
            crate::player::metrics::inc_sessions_created();
            PlayerSession { current_index: 0, last_activity: now }
        })
        .current_index
}

the fast path avoids taking any exclusive lock by using DashMap's read-only get(). only new sessions hit the entry API which needs an exclusive shard lock. since most requests are from existing sessions the fast path is the common case.

two TTLs in cleanup_stale_sessions

the cleanup task runs hourly and applies different rules by session type:

  • player sessions — 24 hours: every range request during playback touches last_activity, so an active listener is never cleaned up. 24 hours lets someone leave a tab open overnight and come back.
  • broadcaster sessions — 30 seconds: heartbeats arrive every 2 seconds normally. 30 seconds of silence means 15 missed heartbeats — unambiguously a dead connection, not a momentary hiccup.

rate limiter buckets idle for over an hour are also pruned so the DashMap doesn't grow forever from IPs that connected once.

templates.rs + player.html (Askama)rust

Askama is a compile-time template engine. it reads HTML files at build time and generates Rust code. by the time the binary runs, template rendering is just string formatting.

what Askama generates

when you run cargo build, Askama reads player.html and generates a Rust impl roughly like this:

by the time the binary runs there is no file reading, no template parsing — it's all inlined Rust code that formats a string. the {% if is_video %} block in player.html hides the audio element and adds the video source on the server side before anything reaches the browser.

IntoResponse sets the session cookie

        match self.render() {
            Ok(html) => Response::builder()

after the HTML arrives, the browser parses it and encounters <link> and <script> tags. each triggers an independent HTTP GET. the router matches these to the ServeDir mounts and serves the files. defer scripts all download in parallel but execute in document order after parsing completes.

Bluetooth mode

Bluetooth mode is an optional setting that briefly mutes audio during seeks. some Bluetooth codecs re-negotiate when audio is interrupted mid-stream, producing a harsh click. muting during the seek and unmuting 1 second after playing fires covers the codec re-negotiation window. the setting is persisted in localStorage so it survives page reload.

the seek-aware play() patch

there is a race condition in the HTML5 media API: if you set src, call load(), set currentTime, and immediately call play(), the play often fails with an AbortError because the seek is still pending. the fix is to defer play() until the seeked event fires:

function patchPlay(el, originalPlay) {
    el.play = () => {
        if (!seekPending) return originalPlay();
        return new Promise((resolve, reject) => {
            el.addEventListener("seeked",
                () => originalPlay().then(resolve).catch(reject),
                { once: true });
        });
    };
}
patchPlay(audioEl, originalPlayAudio);
patchPlay(videoEl, originalPlayVideo);  // both elements, switch can happen any time

shared event handlers on both elements

wireMediaEvents(el) is called for both audioEl and videoEl. the handlers are identical because behavior should be the same regardless of which element is active:

  • seeking — sets seekPending = true, defers Bluetooth mute via setTimeout(0) so currentTime has updated first
  • seeked — clears seekPending
  • playing — calls btUnmuteOnPlaying() (1s delayed unmute if BT mode)
  • ended — three branches: radio mode waits for AutoNext/Sync; broadcasting sends AutoNext then calls playNext(); private mode just plays next
  • timeupdate — updates the radio progress bar and time display if in radio mode
  • play — checks session validity (401 → pause + alert), updates OS media session metadata
  • loadstart — calls playlistManager.updateCurrentFromAudioSrc() to keep playlist highlight in sync

loadCustomOrder — restoring from localStorage

loadCustomOrder() {
    const savedOrder = localStorage.getItem(this.storageKey);  // "playlist_order_{sessionId}"
    if (!savedOrder) { debugLog("No custom order, using default"); return; }

    const orderIds = JSON.parse(savedOrder);
    // filter IDs that no longer exist (deleted files)
    const validIds = orderIds.filter((id) => this.medias.some((s) => s.id === id));
    const orderedMedias = validIds.map((id) => this.medias.find((s) => s.id === id));
    // append any new server files not in saved order
    this.medias.forEach((m) => { if (!validIds.includes(m.id)) orderedMedias.push(m); });
    this.medias = orderedMedias;
}

the storage key includes the session ID so each browser tab has its own independent ordering. the validation step removes deleted files cleanly and newly uploaded files appear at the end.

syncCurrentFromAudio — reading back what the server set

syncCurrentFromAudio() {
    const el = window._activeMedia ?? document.getElementById("audio-player");
    const rawSrc = el.currentSrc || el.src || "";
    // audio.src on an element with a <source> child returns the document URL

handlers.rs — radio_websocket()rust

the server-side WebSocket upgrade endpoint. validates IP rate limit, session ID format, and session existence before upgrading. this is the gatekeeper — it never does radio work itself.

pub async fn radio_websocket(
    ws: WebSocketUpgrade,
    State(state): State<SharedState>,
    headers: HeaderMap,
    ConnectInfo(addr): ConnectInfo<PeerAddr>,
) -> impl IntoResponse {
    let ip = headers.get("x-real-ip")
        .and_then(|v| v.to_str().ok())
        .map(|s| s.to_string())
        .unwrap_or_else(|| addr.0.ip().to_string());

    // 1. rate limit by IP
    if let Err(_) = state.ws_rate_limiter.check_and_consume(&ip) {
        return (StatusCode::TOO_MANY_REQUESTS, "Too many connections").into_response();
    }

    // 2. validate session ID is a UUID
    let validated_session_id = match SessionId::new(session_id_str) {
        Ok(id) => id,
        Err(e) => return (StatusCode::BAD_REQUEST, "Invalid session ID").into_response(),
    };

    // 3. session must exist on the server
    if !state.sessions.contains_key(validated_session_id.as_str()) {
        return (StatusCode::UNAUTHORIZED, "Session expired").into_response();
    }

    // 4. upgrade — hand off to handle_radio_connection
    ws.on_upgrade(|socket| handle_radio_connection(socket, state, validated_session_id.into_inner(), ip))
}

the checks are ordered deliberately: rate limiting is cheapest (one DashMap lookup), so it runs first. UUID validation is CPU-only (no I/O), so it runs second. session existence requires a DashMap lookup, so last. if any check fails the connection is rejected before the socket is upgraded — the TCP connection is closed before any WebSocket handshake completes.

radio.rs — handle_radio_connection()rust

the two-task architecture for a WebSocket session: a receive task parses and routes inbound messages, a send task drains three channel sources and writes outbound frames. they communicate through two narrow channels.

splitting the socket and the two channels

let (mut sender, mut receiver) = socket.split();
let (out_tx, mut out_rx)   = mpsc::channel::<RadioMessage>(32);
let (tuned_tx, mut tuned_rx) = tokio::sync::watch::channel::<Option<String>>(None);

the socket is split because you can't have two tasks holding the same socket — split gives exclusive ownership of each half. out_tx/out_rx is mpsc (multi-producer, single-consumer) — the receive task uses it to send replies back to this specific client through the send task. tuned_tx/tuned_rx is a watch channel — always holds one value, notifies the receiver on change. the receive task writes the current broadcaster ID here; the send task watches for changes and swaps subscriptions.

the send task's biased select!

tokio::select! {
    biased;  // check branches in order, not randomly

    // 1. broadcaster's channel — hottest, checked first
    Ok(msg) = async {
        match &mut current_tuned_broadcaster_rx {
            Some(rx) => rx.recv().await,
            None     => std::future::pending().await,  // never ready if not tuned
        }
    } => { send_prepared(&mut sender, &msg).await }

    // 2. global announcements
    Ok(msg) = global_broadcast_rx.recv() => { send_prepared(&mut sender, &msg).await }

    // 3. direct replies from receive task
    Some(msg) = out_rx.recv() => { send_message(&mut sender, &msg).await }

    // 4. tuning change
    Ok(()) = tuned_rx.changed() => {
        match tuned_rx.borrow().clone() {
            Some(id) => {
                let tx = state_clone.broadcast_channels.get(&id).map(|r| r.clone());
                current_tuned_broadcaster_rx = tx.map(|t| t.subscribe());
            }
            None => { current_tuned_broadcaster_rx = None; }
        }
    }
}

biased means branches are evaluated in order rather than randomly when multiple are ready. the broadcaster's sync channel is the busiest path — every seek fires through it. having it first minimizes latency on the critical path. std::future::pending() returns a future that never resolves, effectively removing that branch from the select without any special casing.

watch channel drives dynamic subscription

when the receive task processes TuneIn, it calls tuned_tx.send(Some(broadcaster_id)). the send task's tuned_rx.changed() branch fires, looks up the broadcaster's broadcast::Sender in broadcast_channels, calls .subscribe() to get a receiver, and stores it as current_tuned_broadcaster_rx. from that moment every Sync the broadcaster generates goes directly to this listener's send loop. TuneOut sends None, setting current_tuned_broadcaster_rx = None, which makes the first branch eternally pending.

    _ = &mut receive_task => { send_task.abort(); }
}

Playing Media

a user clicks play or the page loads with audio already set. the browser starts requesting bytes. if the user seeks to an unbuffered region the browser issues an HTTP range request.

parsing the Range header

fn parse_range_header(headers: &HeaderMap, file_size: u64) -> Option<(u64, u64)> {
    let range_str = headers.get(header::RANGE)?.to_str().ok()?;
    let range_str = range_str.strip_prefix("bytes=")?;
    let (start_str, end_str) = range_str.split_once('-')?;
    let start: u64 = start_str.parse().ok()?;
    let end: u64 = if end_str.is_empty() {
        file_size - 1    // "bytes=2M-" means from 2M to end of file
    } else {
        end_str.parse::<u64>().ok()?.min(file_size - 1)
    };
    Some((start, end))
}

every ? returns None on failure. callers that receive None serve the full file. the .min(file_size - 1) clamps the end so a client requesting beyond EOF gets the file end rather than an error.

the MP4 moov atom problem — why always 206
MP4 files have a metadata block (the "moov atom") that contains duration, codec info, and sample offsets. when this atom is at the end of the file (the default from most encoders), the browser must download the entire file before it can begin playback. a plain 200 OK response triggers this path in some browsers. responding with 206 Partial Content — even for a full-file response — signals range support, which causes the browser to instead issue targeted range requests for just the moov atom, enabling near-instant playback start. ffmpeg's -movflags +faststart moves the moov atom to the front on upload, but the 206 path handles files that weren't re-encoded. 
// if no range: respond 206 with full-file Content-Range anyway
Ok(Response::builder()

    .header(header::CONTENT_RANGE, format!("bytes 0-{}/{}", file_size-1, file_size))
    .header(header::CACHE_CONTROL, "public, max-age=31536000")  // 1 year cache
    .body(body).unwrap())

1 MB streaming chunks

const STREAMING_CHUNK_BYTES: usize = 1 * 1024 * 1024;
let stream = ReaderStream::with_capacity(file, STREAMING_CHUNK_BYTES);

at most 1 MB of any file lives in RAM at once, regardless of file size. Axum streams the body directly from the ReaderStream — each chunk is read from disk, sent over the socket, and dropped.

aggressive cache headers

Cache-Control: public, max-age=31536000 — one year. media files are immutable once uploaded. the browser caches every byte range it fetches. a user who has played through a file once and seeks back costs zero network traffic the second time.

session activity touch on range requests

every range request touches session.last_activity. this is how active listeners stay alive in the stale session cleanup — they're constantly requesting bytes, and each request refreshes their TTL without any extra mechanism.

Latency compensation — both legs

staying in sync requires compensating for two network legs: broadcaster to server, and server to listener.

BROADCASTER SERVER LISTENER Date.now() = T1 BroadcastUpdate ──────────────▶ server receives at T2 latency_ms = T2 - T1 adjusted = playback_time + (latency_ms/1000) Sync(adjusted) ──────────────────────────▶ arrives at T3 ON TUNEIN (one-time): TuneIn sent at _tuneInSentAt initial Sync(adjusted) ───────────────────▶ elapsed = T3 - tuneInSentAt final = adjusted + elapsed

leg 1 — broadcaster to server

the broadcaster stamps client_timestamp_ms = Date.now() on every outgoing message. the server subtracts it from its own wall clock on receipt. this is approximate — it includes both send and receive times conflated as one-way latency — but with similar jitter in both directions it's a reasonable estimate. the result is added to playback_time in the outgoing Sync so listeners aim slightly ahead of where the broadcaster was when the update was recorded.

leg 2 — server to listener (TuneIn only)

if (this._tuneInSentAt !== null) {
    const elapsed = (Date.now() - this._tuneInSentAt) / 1000;
    this._tuneInSentAt = null;
    const adjusted = playback_time + elapsed;
    if (adjusted < duration) playback_time = adjusted;
    debugLog(`TuneIn latency compensation: +${elapsed.toFixed(3)}s`);
}

this only runs on the first Sync after TuneIn. subsequent Syncs don't need this because the listener's element is already playing and advancing on its own — small drift is absorbed naturally and the heartbeat keeps positions fresh.

_loadingMediaIndex — preventing duplicate canplay listeners

the race condition

imagine the broadcaster switches to a new track. the listener receives a Sync with the new media index. without the guard:

  1. Sync 1 arrives. listener sees different media index, sets audio.src, calls load(), attaches a canplay listener.
  2. Sync 2 arrives before canplay fires (broadcaster seeked or network delivered them together). listener sees media index again, sets audio.src again, calls load() again, attaches a second canplay listener.
  3. canplay fires. both listeners run. listener 1 seeks to position A and starts playing. listener 2 immediately seeks to position B and starts playing again. jump, stutter, wrong position.

the guard

if (this._loadingMediaIndex !== media_index) {
    // case 1: start a fresh load
    this._loadingMediaIndex = media_index;
    this.audio.src = ...;
    this.audio.load();
    this.audio.addEventListener("canplay", () => {
        this._loadingMediaIndex = null;       // ← clear on completion
        // use _pendingSeekTime (latest value from any subsequent Syncs)
        this.audio.currentTime = this._pendingSeekTime;
        if (this._pendingIsPlaying) this._radioPlay();
    }, { once: true });
} // else: _loadingMediaIndex === media_index → already loading → skip, just update _pendingSeekTime

when the second Sync arrives for the same media index, _loadingMediaIndex already matches. the code skips the load and only updates _pendingSeekTime with the newer position. the already-attached single canplay listener uses that updated value when it fires. one load, one listener, correct position.

the only time it abandons a load and starts fresh is if the broadcaster switches to a completely different track while the previous one is still loading. then media_index differs from _loadingMediaIndex and a new load begins, discarding the previous one.

DeleteOnDrop — RAII temp file cleanup

struct DeleteOnDrop(Option<std::path::PathBuf>);
impl Drop for DeleteOnDrop {
    fn drop(&mut self) {
        if let Some(p) = self.0.take() {
            std::fs::remove_file(&p).ok();  // ignore errors — best effort
        }
    }
}
fn disarm(&mut self) -> std::path::PathBuf { self.0.take().unwrap() }

RAII: tie the lifetime of a resource to an object. when the object is dropped (goes out of scope, early return, panic), the resource is released. here "release" means "delete the temp file". the disarm() method takes the path out of the Option, leaving None, so the Drop impl does nothing — ownership of cleanup is transferred to whoever received the path.

pub async fn cleanup_stale_sessions(state: Arc<AppState>) {
    let mut interval = tokio::time::interval(Duration::from_secs(3600));
    loop {
        interval.tick().await;  // first tick fires immediately, then every 1h
        let now = std::time::Instant::now();

        // pass 1: player sessions — 24h TTL
        state.sessions.retain(|session_id, session| {
            now.duration_since(session.last_activity).as_secs() < 24 * 3600
        });

        // pass 2: rate limiter buckets — 1h TTL
        state.ws_rate_limiter.cleanup_old_buckets();

        // pass 3: stale broadcasters — 30s TTL
        let stale: Vec<String> = state.broadcast_states.iter()
            .filter(|e| {
                let age_ms = now_ms().saturating_sub(e.value().server_timestamp_ms);
                age_ms / 1000 > 30
            })
            .map(|e| e.key().clone())
            .collect();

        for broadcaster_id in stale {
            state.broadcast_states.remove(&broadcaster_id);
            state.broadcaster_listeners.remove(&broadcaster_id);
            // notify all clients with BroadcasterOffline
            let offline_msg = Arc::new(PreparedMessage::new(&RadioMessage::BroadcasterOffline {
                broadcaster_id: broadcaster_id.clone(),
            }));
            state.global_broadcast_tx.send(offline_msg).ok();
            state.broadcast_channels.remove(&broadcaster_id);
        }
        broadcast_analytics(&state);
    }
}

DashMap::retain is a concurrent in-place filter — it acquires each shard lock, removes entries that don't satisfy the predicate, and releases. it runs once per shard so lock contention is brief. the broadcast cleanup collects IDs first (to avoid holding DashMap refs across awaits), then removes them one by one and sends BroadcasterOffline for each.

error.rsrust

the PlayerError enum. implements IntoResponse so Axum converts errors to HTTP responses automatically. #[from] attributes auto-generate conversions from std errors so handlers can use ? freely.

#[derive(Error, Debug)]
pub enum PlayerError {
    #[error("Invalid session ID format: {0}")]
    InvalidSessionId(String),

    #[error("IO error: {0}")]
    IoError(#[from] std::io::Error),     // #[from] generates: impl From<io::Error> for PlayerError

    #[error("JSON serialization failed: {0}")]
    SerializationError(#[from] serde_json::Error),
    // ...
}

impl IntoResponse for PlayerError {
    fn into_response(self) -> Response {
        let (status, message) = match self {
            PlayerError::InvalidSessionId(_)  => (StatusCode::BAD_REQUEST,       self.to_string()),
            PlayerError::BroadcasterNotFound(_) => (StatusCode::NOT_FOUND,       self.to_string()),
            PlayerError::RateLimitExceeded(_) => (StatusCode::TOO_MANY_REQUESTS, self.to_string()),
            PlayerError::BroadcastUnauthorized(_) => (StatusCode::UNAUTHORIZED,  self.to_string()),
            _                                 => (StatusCode::INTERNAL_SERVER_ERROR, "Internal server error".to_string()),
        };
        (status, message).into_response()
    }
}

pub type PlayerResult<T> = Result<T, PlayerError>;

the #[from] attribute on IoError auto-generates a From<std::io::Error> for PlayerError impl. code doing file I/O can use ? and the error is wrapped automatically. handlers return Result<T, PlayerError>; Axum calls into_response() on the error, producing the right HTTP status code without any manual response building. PlayerResult<T> is an alias so you don't write the full type everywhere.

rate_limit.rsrust

token bucket rate limiter. each key gets a bucket with a capacity and a refill rate. tokens refill continuously; each request consumes one; empty bucket = rejected.

pub fn check_and_consume(&self, session_id: &str) -> PlayerResult<()> {
    let mut bucket = self.buckets.entry(session_id.to_string())
        .or_insert_with(|| TokenBucket { tokens: self.capacity, last_refill: Instant::now() });

    let elapsed = Instant::now().duration_since(bucket.last_refill).as_secs_f64();
    bucket.tokens = (bucket.tokens + elapsed * self.refill_rate).min(self.capacity);
    bucket.last_refill = Instant::now();

    if bucket.tokens >= 1.0 { bucket.tokens -= 1.0; Ok(()) }
    else                     { Err(PlayerError::RateLimitExceeded(session_id.to_string())) }
}

DashMap is used rather than RwLock<HashMap> because check_and_consume always writes (it updates tokens and last_refill). a RwLock would give no read-concurrency benefit and its write lock would serialize all rate limit checks across sessions. DashMap shards mean concurrent sessions contend only within the same shard.

validation.rsrust

the SessionId newtype. parse-don't-validate applied to UUIDs. functions accepting SessionId carry a compile-time guarantee that UUID validation has already passed.

pub struct SessionId(String);  // newtype — a validated UUID in a named wrapper

impl SessionId {
    pub fn new(id: String) -> PlayerResult<Self> {
        uuid::Uuid::parse_str(&id)
            .map_err(|_| PlayerError::InvalidSessionId(format!(
                "Session ID must be a valid UUID, got: {}", id
            )))?;
        Ok(SessionId(id))
    }
    pub fn as_str(&self) -> &str { &self.0 }
    pub fn into_inner(self) -> String { self.0 }
}

parse-don't-validate: instead of validating a String everywhere it's used, convert it once into a type that can only be constructed through validation. any function accepting a SessionId parameter carries a compile-time guarantee that UUID validation passed — no re-checking needed deeper in the call chain. if you add a function that should only receive valid session IDs, just add a SessionId parameter and the compiler enforces it.

Design Decisions

decisionrationale
no databasesessions are ephemeral (24h max), broadcast state is in-memory by nature. the only persistence is the media folder and localStorage. SQLite would add migrations, dependencies, and WAL maintenance for no benefit.
no frontend frameworktwo interactive classes, six auxiliary files. framework abstraction overhead would exceed the complexity it manages.
always 206 Partial Contenta plain 200 causes some browsers to abort and restart when seeking, adding a multi-second delay before MP4 playback. 206 signals range support universally and avoids the moov atom download-all-first path.
DashMap everywhere (except playlist)all hot paths hit different session keys. sharded concurrency eliminates the global write bottleneck of a single RwLock.
PreparedMessage + Arcserialize once per broadcast event. O(1) per listener (pointer clone) instead of O(n) serializations.
biased select! in send taskthe broadcaster's sync channel is always the highest-priority source. checking it first reduces end-to-end seek-to-listener latency.
video conversion at upload timeconverting on every stream request would saturate CPUs and add first-byte latency. converting once at upload gives instant subsequent streaming.
DeleteOnDrop for temp filesasync error handling is subtle — an early return due to a client disconnect would skip cleanup at the end of a function. RAII binds cleanup to object lifetime, which Rust guarantees regardless of exit path.
TCP_NODELAY on every socketNagle's algorithm batches small writes for bulk efficiency. WebSocket sync frames are tiny and latency-sensitive. disabling batching costs negligible bandwidth and noticeably reduces seek-to-sync delay on the critical path.
admin panel is publicdesigned for a trusted home network. session IDs are non-guessable UUIDs. for multi-tenant deployment this would need authentication.
_loadingMediaIndex guardprevents the double canplay listener race that causes seek jumps on high-latency connections or when the broadcaster seeks rapidly.
AutoNext vs Sync for track transitionsSync would force immediate track switch, cutting the listener's last few seconds. AutoNext lets listeners finish their copy of the current track, making transitions smooth.
per-file errors in uploadone bad file in a ten-file batch should not abort the other nine. collecting errors and returning them allows partial success with clear per-file feedback.
