stargzr

a self-hosted media player with live radio broadcasting — built from scratch in Rust and vanilla JavaScript, with no database and no framework.

Rust · Axum · Tokio Vanilla JavaScript WebSockets HTTP range streaming
00 — introduction

What is this?

stargzr is a web app that runs from a single Rust binary. You point it at a folder of audio and video files, and it gives everyone on your network a browser-based media player. Any user can start broadcasting — sharing their current playback state over a WebSocket — and other users can tune in and hear exactly what the broadcaster is hearing, synchronized in real time. Think of it as a private radio station where you can see who is playing what and jump into their stream.

the architecture below shows the two main flows. private mode (left) is a straightforward client-server media player. radio mode (right) adds a real-time sync layer on top.

PRIVATE MODE RADIO MODE Browser A Browser B (broadcaster) │ GET /stream/id/{id} │ BroadcastUpdate (WebSocket) │ Range: bytes=… │ ▼ ▼ Rust Server Rust Server │ 206 Partial Content │ serialise once → Arc<PreparedMessage> │ 1MB chunks │ fan out through broadcast channel ▼ ▼ Browser A plays Browser C (listener) Browser D (listener) Browser E (listener) each seeks to broadcaster's position

the interesting engineering lives in the gap between those two sides: compensating for network latency so listeners stay in sync, recovering state after a disconnection, handling video versus audio with the same code path, and making it all fast enough that a seek feels instant even with many listeners.

01 — backend

The Rust Server

the server is built on Axum, a web framework built on Tokio (async runtime) and Hyper (HTTP). A few Rust concepts worth knowing before diving in:

conceptwhat it means
Arc<T>atomically reference-counted pointer. cheap to clone; lets multiple threads share the same data without copying it.
RwLock<T>multiple readers or one writer at a time. readers never block each other.
DashMap<K,V>concurrent hash map. internally sharded so operations on different keys rarely contend.
.awaitsuspend this async task until the future resolves; yield the thread to other tasks.
Result<T, E>either success (Ok(T)) or failure (Err(E)). the ? operator returns early on error.
types.rs rust

defines every shared data structure: the application state, all WebSocket message variants, and the media/session/broadcast types. everything else imports from here.

AppState — the single source of truth

every handler function receives a clone of Arc<AppState>. cloning an Arc only increments a counter — the underlying data is not copied. this is the standard Rust pattern for sharing state across threads without locks at the top level.

pub struct AppState {
    pub playlist:           Arc<RwLock<Vec<MediaInfo>>>,
    pub sessions:           DashMap<String, PlayerSession>,
    pub broadcast_states:   DashMap<String, BroadcastState>,
    pub broadcast_channels: DashMap<String, broadcast::Sender<Arc<PreparedMessage>>>,
    pub broadcaster_listeners: DashMap<String, HashSet<String>>,
    pub session_tuned_to:   DashMap<String, String>,
    pub global_broadcast_tx: broadcast::Sender<Arc<PreparedMessage>>,
    pub active_connections: AtomicUsize,
    pub last_analytics_ms:  AtomicU64,
    pub ws_rate_limiter:    RateLimiter,
    pub upload_quotas:      DashMap<String, u64>,
    pub conversion_semaphore: Arc<Semaphore>,
}

why DashMap instead of RwLock<HashMap>?

a RwLock<HashMap> serialises every write across the entire map. a heartbeat from broadcaster A would block a TuneIn from listener B even though they touch completely different keys. DashMap shards internally — by default into roughly 4× the number of CPU cores — so operations on different keys almost never block each other.

why is playlist wrapped differently?

the playlist is behind RwLock<Vec> rather than DashMap because it is a sequential structure accessed by integer index. the only time an exclusive write lock is needed is during uploads (inserting a new entry in sorted position). everything else — streaming, playlist fetches, radio sync — needs only a read lock, and many read locks can be held simultaneously.

PreparedMessage — serialize once, fan out to N listeners

pub struct PreparedMessage {
    pub json: String,
}
impl PreparedMessage {
    pub fn new(msg: &RadioMessage) -> Self {
        Self { json: serde_json::to_string(msg).unwrap() }
    }
}

when a broadcaster seeks, the server needs to push a Sync message to potentially hundreds of listeners. serialising the same struct once per listener would be wasteful. instead the server serialises once into a PreparedMessage, wraps it in Arc, and sends that arc through the broadcast channel. each listener's send task receives a clone of the arc — which is just a pointer increment — and writes the same bytes to their socket. this is zero-copy fan-out.

the RadioMessage enum

all WebSocket messages are variants of a single RadioMessage enum. Serde's #[serde(tag = "type")] attribute serialises the variant name as a "type" field in JSON:

#[derive(Serialize, Deserialize)]
#[serde(tag = "type")]
pub enum RadioMessage {
    Sync         { broadcaster_id: String, media_index: usize, playback_time: f64, is_playing: bool, server_timestamp_ms: u128 },
    Heartbeat    { broadcaster_id: String, playback_time: f64, client_timestamp_ms: u64 },
    TuneIn       { broadcaster_id: String },
    TuneOut,
    BroadcastUpdate { broadcaster_id: String, media_index: usize, playback_time: f64, is_playing: bool, client_timestamp_ms: u64 },
    StartBroadcasting { .. },
    StopBroadcasting  { .. },
    BroadcasterOnline  { broadcaster_id: String },
    BroadcasterOffline { broadcaster_id: String },
    AutoNext     { broadcaster_id: String, next_media_index: usize, server_timestamp_ms: u64 },
    Analytics    { .. },
    QueryBroadcastState   { session_id: String },
    BroadcastStateResponse { .. },
    ServerShutdown { message: String },
    Error          { message: String },
}

on the wire this becomes {"type":"TuneIn","broadcaster_id":"...uuid..."}. the JavaScript frontend uses the same type field in its switch statement.

why a single enum?
having all messages as one type means the compiler can enforce that every variant is handled in match blocks. if you add a new message type, every match that doesn't handle it becomes a compile error. no forgotten cases at runtime.

BroadcastState — what the server knows about a broadcaster

pub struct BroadcastState {
    pub broadcaster_id:          String,
    pub media_index:             usize,
    pub media_name:              String,
    pub playback_time:           f64,   // raw, as reported by broadcaster
    pub is_playing:              bool,
    pub server_timestamp_ms:     u128,  // when this was recorded
    pub listener_count:          usize,
    pub transmission_latency_ms: u64,   // estimated one-way lag
}

transmission_latency_ms is computed from the client_timestamp_ms field each broadcaster stamps on every outgoing message. subtracting it from the server's wall clock gives the one-way transit time. this is stored on the state so that late-joining listeners can receive an already-adjusted position via TuneIn.

session.rs rust

manages browser sessions (via cookies), tracks per-user playlist position, and runs the hourly cleanup task that removes stale sessions.

how sessions are created

when a browser loads the player page, the server looks for a player_session cookie. if absent, a UUID v4 is generated and set as a cookie in the HTTP response. from then on every request from that tab carries that UUID.

pub fn get_session_id(headers: &HeaderMap) -> String {
    headers.get(header::COOKIE)
        .and_then(|v| v.to_str().ok())
        .and_then(|cookies| {
            cookies.split(';').find_map(|c| {
                c.trim().strip_prefix("player_session=")
            })
        })
        .map(|s| s.to_string())
        .unwrap_or_else(|| Uuid::new_v4().to_string())
}

there is no login, no password, no database lookup. the cookie is simply a random identifier. this works because all users of stargzr are assumed to be on the same trusted network.

get_or_create_position — the DashMap entry API

pub fn get_or_create_position(state: &AppState, session_id: &str) -> usize {
    // fast path — read without taking an exclusive lock
    if let Some(session) = state.sessions.get(session_id) {
        return session.current_index;
    }
    // slow path — create entry if it doesn't exist yet
    let mut session = state.sessions
        .entry(session_id.to_string())
        .or_insert_with(|| PlayerSession {
            current_index: 0,
            last_activity: std::time::Instant::now(),
        });
    session.last_activity = std::time::Instant::now();
    session.current_index
}

the fast path avoids taking any exclusive lock by using DashMap's read-only get(). only new sessions hit the entry path which needs to acquire a shard write lock.

cleanup_stale_sessions — two different TTLs

the cleanup task wakes every hour and applies different rules to different kinds of sessions:

  • player sessions — kept alive for 24 hours of inactivity. range requests during playback automatically touch last_activity, so an active listener never gets cleaned up.
  • broadcaster sessions — considered stale after 30 seconds without a heartbeat. heartbeats fire every 2 seconds normally, so 30 seconds allows several missed beats before declaring the broadcaster gone.
note
custom playlist order is stored in localStorage in the browser, not on the server. so even if a session is cleaned up, reopening the page will restore the user's playlist order. only the current track position resets.
handlers.rs rust

all HTTP handlers: serving the player page, media streaming with range support, file/folder downloads, uploads with ffmpeg video conversion, and the admin state endpoint.

HTTP range requests (RFC 7233)

browsers do not download audio/video files completely before playing. they issue range requests: Range: bytes=2000000-3000000. this is what makes seeking work — the browser asks for exactly the bytes it needs and the server delivers only those bytes.

fn parse_range_header(headers: &HeaderMap, file_size: u64) -> Option<(u64, u64)> {
    let range_str = headers.get(header::RANGE)?.to_str().ok()?;
    let range_str = range_str.strip_prefix("bytes=")?;
    let (start_str, end_str) = range_str.split_once('-')?;
    let start: u64 = start_str.parse().ok()?;
    let end: u64 = if end_str.is_empty() {
        file_size - 1                          // "bytes=2M-" means from 2M to end
    } else {
        end_str.parse::<u64>().ok()?.min(file_size - 1)
    };
    Some((start, end))
}

when a range is present the server seeks the file handle to start, wraps it in a ReaderStream limited to end - start + 1 bytes, and responds with HTTP 206 Partial Content.

why always 206, even for a full-file response?

when no range header is present — the browser's first request — the server still responds with 206 and a full-file Content-Range: bytes 0-N/N header, rather than the more natural 200.

the MP4 moov atom problem
MP4 files store a metadata block called the moov atom that contains duration, codec info, and sample offsets. when this atom is at the end of the file (the default from most encoders), the browser must download the entire file before it can begin playback. a plain 200 response triggers this path. responding with 206 signals that this server supports range requests, which causes the browser to skip ahead and issue targeted range requests for the moov atom, enabling near-instant playback start. ffmpeg's -movflags +faststart flag moves the moov atom to the front, but the 206 path handles files that weren't encoded with faststart too.

streaming in 1 MB chunks

const STREAMING_CHUNK_BYTES: usize = 1 * 1024 * 1024;
let stream = ReaderStream::with_capacity(file, STREAMING_CHUNK_BYTES);

at most 1 MB of any file lives in RAM at once, regardless of file size. Axum streams the body directly from the ReaderStream — each chunk is read from disk, sent over the socket, and then dropped.

aggressive cache headers

.header(header::CACHE_CONTROL, "public, max-age=31536000")

one year of caching. because media files never change (they are immutable once uploaded), the browser can cache every byte range it fetches. a user who has already played a file and seeks back through it costs zero bytes of network traffic the second time.

video upload & ffmpeg conversion

uploaded videos are immediately converted to H.264/AAC MP4 so they work in every browser. the ffmpeg command is chosen carefully:

ffmpeg -i input.mkv \
  -map 0:v:0 -map 0:a:0 \
  -c:v libx264 -crf 23 -maxrate 2M -bufsize 4M \
  -c:a aac -b:a 192k -ac 2 \
  -movflags +faststart \
  output.mp4 \
  -map 0:s:0 -c:s webvtt output.vtt
flagwhat it does
-crf 23constant quality factor. lower = higher quality and larger file. 23 is a good all-round default.
-maxrate 2M -bufsize 4Mcaps bitrate spikes that cause buffer underruns during HTTP streaming
-ac 2force stereo. AAC 5.1 surround has a known bug (PCE issue) causing silent playback in Chrome/Firefox
-movflags +faststartmoves the moov atom to the front of the file. enables playback before the full file downloads.
-c:s webvtt output.vttextracts embedded subtitles as a separate .vtt file. MP4 cannot embed WebVTT natively.

only one conversion runs at a time, enforced by a Semaphore with a count of 1. this prevents CPU saturation on low-power servers when multiple users upload videos simultaneously.

DeleteOnDrop — RAII temp file cleanup

when zipping a folder for download, the server writes to a temp file. if anything fails partway through — disk error, client disconnects, code bug — the temp file must not be left behind.

struct DeleteOnDrop(Option<std::path::PathBuf>);
impl Drop for DeleteOnDrop {
    fn drop(&mut self) {
        if let Some(p) = self.0.take() {
            std::fs::remove_file(&p).ok();
        }
    }
}
// disarm: take ownership without triggering cleanup
fn disarm(&mut self) -> std::path::PathBuf { self.0.take().unwrap() }

RAII (resource acquisition is initialization) is a Rust/C++ pattern: tie the lifetime of a resource to an object. when the object is dropped (goes out of scope, returns early, panics), the resource is released. here "release" means "delete the temp file". the disarm() method hands ownership to the streaming side, which installs its own guard to delete the file once the last byte is sent.

radio.rs rust

the heart of stargzr. manages each WebSocket connection using two concurrent tasks, routes all radio messages, fans out sync updates to listeners, and handles latency compensation.

the two-task architecture

each WebSocket connection spawns two Tokio tasks that run concurrently:

  • receive task — reads frames from the socket, deserialises them, calls handle_client_message
  • send task — waits on three channel sources and writes frames to the socket

they communicate through:

  • out_tx / out_rx — an mpsc channel from receive→send for direct replies (e.g., the initial Sync on TuneIn)
  • tuned_tx / tuned_rx — a watch channel that carries the broadcaster ID the listener is currently tuned to
let (out_tx, mut out_rx)  = mpsc::channel::<RadioMessage>(32);
let (tuned_tx, mut tuned_rx) = tokio::sync::watch::channel::<Option<String>>(None);
let mut global_broadcast_rx = state.global_broadcast_tx.subscribe();

the send task's biased select!

tokio::select! {
    biased;  // ← check branches in order, not randomly

    // 1. broadcaster's channel — hottest path, checked first
    Ok(msg) = async {
        match &mut current_tuned_broadcaster_rx {
            Some(rx) => rx.recv().await,
            None => std::future::pending().await,  // never ready if not tuned
        }
    } => { send_prepared(&mut sender, &msg).await }

    // 2. global announcements (BroadcasterOnline, Analytics, Shutdown)
    Ok(msg) = global_broadcast_rx.recv() => {
        send_prepared(&mut sender, &msg).await
    }

    // 3. direct replies from the receive task
    Some(msg) = out_rx.recv() => {
        send_message(&mut sender, &msg).await
    }

    // 4. broadcaster tuning changed
    Ok(()) = tuned_rx.changed() => { /* swap current_tuned_broadcaster_rx */ }
}

biased skips Tokio's default random branch selection. the tuned broadcaster's channel is the busiest path at runtime (every seek fires there), so having it always checked first reduces latency on the critical path.

when current_tuned_broadcaster_rx is None (not tuned to anyone), the std::future::pending().await inside returns a future that never resolves. the branch effectively disappears from the select without requiring any special-casing outside.

the watch channel for tuning changes

a watch channel always holds the latest value and notifies the receiver whenever it changes. when the receive task processes a TuneIn message it writes the broadcaster ID to tuned_tx. the send task's tuned_rx.changed() branch fires, looks up the broadcaster's per-channel broadcast::Sender, calls .subscribe() on it, and stores the resulting receiver as current_tuned_broadcaster_rx. from that moment on, every Sync the broadcaster triggers goes directly into the send loop.

why mpsc for out_rx but broadcast for the broadcaster channel?
mpsc (multi-producer single-consumer) is point-to-point: one sender, one receiver. it is used for direct replies from receive→send because only one send task should receive each reply.

broadcast is one-to-many: one sender, many independent receivers. it is used for broadcaster sync messages because the same Sync message must reach every listener simultaneously. each listener's send task has its own independent receiver and processes messages at its own pace.

BroadcastUpdate — latency compensation on the server side

// client stamps Date.now() before sending; server subtracts on receipt
let latency_ms = (now_ms() as u64).saturating_sub(client_timestamp_ms);

// only adjust on same-media updates (seeks, play/pause)
// media switches start from 0 intentionally — adding latency would skip the intro
let adjusted_playback_time = if is_same_media {
    playback_time + (latency_ms as f64 / 1000.0)
} else {
    playback_time
};

// store raw playback_time on BroadcastState so TuneIn can apply
// the same compensation consistently for late-joining listeners
state.broadcast_states.insert(broadcaster_id.clone(), BroadcastState {
    playback_time,  // raw, not adjusted
    transmission_latency_ms: latency_ms,
    ..
});

analytics throttling — preventing thundering herds

BroadcastUpdate can fire on every seek frame — potentially many times per second. if each one triggered a full analytics broadcast to every connected client, the overhead would be significant. throttling is applied using an atomic compare-exchange:

pub fn broadcast_analytics_throttled(state: &SharedState) {
    let now  = now_ms() as u64;
    let last = state.last_analytics_ms.load(Relaxed);

    if now.saturating_sub(last) < ANALYTICS_THROTTLE_MS { return; }

    // compare_exchange: only one concurrent caller wins the race
    // the loser finds last != expected and returns is_err(), skipping the broadcast
    if state.last_analytics_ms
           .compare_exchange(last, now, Relaxed, Relaxed)
           .is_ok()
    {
        broadcast_analytics(state);
    }
}

without compare_exchange, many concurrent BroadcastUpdate handlers could all pass the time check simultaneously and all fire analytics broadcasts. the atomic swap ensures only one wins.

ensure_same_session — impersonation prevention

fn ensure_same_session(expected: &str, actual: &str) -> Result<(), PlayerError> {
    (expected == actual).then_some(()).ok_or_else(|| {
        tracing::warn!(session_id = %actual, attempted_as = %expected,
                       "Session impersonation attempt");
        PlayerError::BroadcastUnauthorized("...")
    })
}

the session ID is recorded at WebSocket handshake time. every broadcaster message includes a broadcaster_id field. this check ensures the ID in the message matches the one from the handshake — a client cannot claim to be someone else after connecting.

rate_limit.rs rust

token bucket rate limiter. each session or IP gets a bucket with a capacity and a refill rate. requests consume tokens; empty buckets reject requests.

how a token bucket works

imagine a bucket that can hold capacity tokens. tokens refill at a constant rate. each request costs one token. if the bucket is empty the request is rejected. this allows short bursts (the full bucket) while limiting the sustained rate (the refill rate).

pub struct RateLimiter {
    buckets:     DashMap<String, TokenBucket>,
    capacity:    f64,
    refill_rate: f64,  // tokens per second
}

pub fn check_and_consume(&self, session_id: &str) -> PlayerResult<()> {
    let mut bucket = self.buckets.entry(session_id.to_string())
        .or_insert_with(|| TokenBucket { tokens: self.capacity, last_refill: Instant::now() });

    let elapsed = Instant::now().duration_since(bucket.last_refill).as_secs_f64();
    bucket.tokens = (bucket.tokens + elapsed * self.refill_rate).min(self.capacity);
    bucket.last_refill = Instant::now();

    if bucket.tokens >= 1.0 {
        bucket.tokens -= 1.0;
        Ok(())
    } else {
        Err(PlayerError::RateLimitExceeded(session_id.to_string()))
    }
}
limitercapacityrefillrationale
heartbeat5 tokens0.5/s → 1 per 2sheartbeats are timer-driven; strict limit catches runaway timers
broadcast10 tokens2/sallows seek bursts while limiting sustained spam
websocket5 tokens0.1/s → 1 per 10sper-IP; prevents connection flood attacks

old buckets are cleaned up every hour via cleanup_old_buckets() — any bucket whose last refill was over an hour ago is removed. without this cleanup the DashMap would grow indefinitely as new IPs connect and disconnect.

validation.rs rust

the newtype pattern for session IDs — parse-don't-validate applied to UUIDs.

parse-don't-validate

instead of validating a String wherever it is used, the code converts it into a type that can only be constructed through validation. this is the "parse-don't-validate" principle.

pub struct SessionId(String);  // newtype — just a wrapper

impl SessionId {
    pub fn new(id: String) -> PlayerResult<Self> {
        uuid::Uuid::parse_str(&id)
            .map_err(|_| PlayerError::InvalidSessionId(format!(
                "Session ID must be a valid UUID, got: {}", id
            )))?;
        Ok(SessionId(id))
    }

    pub fn as_str(&self) -> &str { &self.0 }
    pub fn into_inner(self) -> String { self.0 }
}

any function that accepts a SessionId parameter carries a compile-time guarantee that UUID validation has already passed — no defensive re-checking needed further down the call chain. the Rust orphan rule prevents implementing traits on foreign types, which is why the newtype exists instead of using String directly.

the orphan rule
Rust only allows implementing a trait for a type if either the trait or the type is defined in your crate. SessionId is defined here, so you can implement any trait for it — including the Connected trait from Axum used for PeerAddr in mod.rs.
error.rs rust

a custom error enum that implements Axum's IntoResponse — errors become HTTP responses automatically, and the ? operator propagates them up the call stack.

#[derive(Error, Debug)]
pub enum PlayerError {
    #[error("Media index {0} out of bounds (playlist size: {1})")]
    InvalidMediaIndex(usize, usize),

    #[error("IO error: {0}")]
    IoError(#[from] std::io::Error),      // ← auto-convert from std::io::Error

    #[error("JSON serialization failed: {0}")]
    SerializationError(#[from] serde_json::Error),  // ← auto-convert from serde errors
    // ...
}

impl IntoResponse for PlayerError {
    fn into_response(self) -> Response {
        let (status, message) = match self {
            PlayerError::InvalidMediaIndex(_, _) => (StatusCode::NOT_FOUND, self.to_string()),
            PlayerError::RateLimitExceeded(_) => (StatusCode::TOO_MANY_REQUESTS, self.to_string()),
            _ => (StatusCode::INTERNAL_SERVER_ERROR, "Internal error".into()),
        };
        (status, message).into_response()
    }
}

the #[from] attribute on IoError and SerializationError auto-generates a From<std::io::Error> for PlayerError impl. this means any code that calls a file IO function can use ? and the error is automatically wrapped into a PlayerError::IoError. handlers return Result<T, PlayerError>, and Axum calls into_response() on the error automatically — no manual error handling in each handler.

mod.rs rust

TCP_NODELAY — disabling Nagle's algorithm

Nagle's algorithm is a TCP optimisation that batches small outgoing packets together to reduce the number of network segments. it improves bulk throughput but adds latency to small, time-sensitive writes. WebSocket sync messages are tiny JSON frames that must arrive as fast as possible. the server wraps the Tokio TcpListener in a NoDelayListener that sets TCP_NODELAY on every accepted socket:

struct NoDelayListener(tokio::net::TcpListener);

impl axum::serve::Listener for NoDelayListener {
    async fn accept(&mut self) -> (Self::Io, Self::Addr) {
        loop {
            match self.0.accept().await {
                Ok((stream, addr)) => {
                    stream.set_nodelay(true).ok();  // ← disable Nagle
                    return (stream, addr);
                }
                Err(e) => tracing::error!("Accept error: {}", e),
            }
        }
    }
}

graceful shutdown

when the server receives SIGTERM (Docker stop, systemctl stop) or SIGINT (Ctrl+C), it does not just kill connections. it:

  1. sends a ServerShutdown message through the global broadcast channel so every browser knows the server is restarting and will try to reconnect
  2. sends BroadcasterOffline for every active broadcaster so listeners gracefully leave radio mode
  3. sleeps 1 second to give the WebSocket send tasks time to flush those messages before Axum starts draining connections

the orphan rule and PeerAddr

Axum's ConnectInfo extractor lets handlers receive the peer's socket address without relying on headers (which can be spoofed). but implementing Axum's Connected trait for std::net::SocketAddr would violate the orphan rule — both types are foreign. the fix is a local newtype:

#[derive(Clone)]
pub struct PeerAddr(pub std::net::SocketAddr);  // defined here → orphan rule satisfied

impl axum::extract::connect_info::Connected<...> for PeerAddr {
    fn connect_info(target: ...) -> Self { PeerAddr(*target.remote_addr()) }
}

radio-player.js javascript

the RadioPlayer class owns the WebSocket connection and all radio-mode logic: connecting, tuning in/out, syncing to a broadcaster, handling reconnections with exponential backoff, and managing wake locks for mobile.

WebSocket lifecycle & reconnection

event handlers are stored in this.boundHandlers because removeEventListener requires the same function reference that was passed to addEventListener. anonymous functions cannot be removed — storing named references solves this.

connectWebSocket() {
    this.ws = new WebSocket(wsUrl);

    const onOpen    = () => { this.reconnectAttempts = 0; /* re-tune if needed */ };
    const onMessage = (event) => { this.handleRadioMessage(JSON.parse(event.data)); };
    const onClose   = async (event) => {
        this.removeWebSocketHandlers();
        if (!this.isIntentionalDisconnect) this.scheduleReconnect();
    };

    // store so removeWebSocketHandlers can find them later
    this.boundHandlers.set("open", onOpen);
    this.boundHandlers.set("close", onClose);
    this.ws.addEventListener("open", onOpen);
    this.ws.addEventListener("close", onClose);
}

reconnection with exponential backoff + jitter

getReconnectDelay() {
    const exponential = this.baseReconnectDelay * Math.pow(2, this.reconnectAttempts);
    const jitter = exponential * 0.2 * (Math.random() - 0.5);
    return Math.min(
        Math.max(this.baseReconnectDelay, exponential + jitter),
        this.maxReconnectDelay
    );
}

jitter is critical: if fifty clients all disconnect simultaneously (server restart) and all reconnect after the same backoff, they flood the server again. randomising the delay spreads the reconnection load across a wider window.

tuneIn — snapshotting private state

before entering radio mode, the player saves everything about the current private playback into preRadioSnapshot:

this.preRadioSnapshot = {
    src:         this.audio.src,
    currentTime: this.audio.currentTime,
    paused:      this.audio.paused,
    mediaId:     window.playlistManager?.currentMediaId,
    isVideo:     snapMedia?.media_type === "video",
};

tuneOut reads this back and restores it exactly — including switching back to the audio or video element that was active, restoring the playback position, and resuming if it was playing. without this, tuning out would reset the user to wherever the browser's element happened to be after following the broadcaster.

_tuneInSentAt — client-side latency compensation

// in tuneIn():
this._tuneInSentAt = Date.now();

// in syncToBroadcaster(), first Sync after TuneIn:
if (this._tuneInSentAt !== null) {
    const elapsed = (Date.now() - this._tuneInSentAt) / 1000;
    this._tuneInSentAt = null;
    const adjusted = playback_time + elapsed;
    if (adjusted < duration) playback_time = adjusted;
}

the server compensates for broadcaster→server latency. this compensates for server→listener latency. together they give the listener a playback time that is close to what the broadcaster is actually at right now, not at the moment the update was recorded.

syncToBroadcaster — three-way logic

syncToBroadcaster(msg) {
    const currentIndex = this.getCurrentMediaIndex();

    if (currentIndex !== media_index || this._loadingMediaIndex === media_index) {
        // case 1 OR 2: wrong track (or already loading this track)
        this._pendingSeekTime   = playback_time;
        this._pendingIsPlaying  = is_playing;

        if (this._loadingMediaIndex !== media_index) {
            // case 1: different track — start loading
            this._loadingMediaIndex = media_index;
            this.audio.src = `/stream/id/${mediaId}`;
            this.audio.load();
            this.audio.addEventListener("canplay", () => {
                this._loadingMediaIndex = null;
                this.audio.currentTime  = this._pendingSeekTime;
                if (this._pendingIsPlaying) this._radioPlay();
            }, { once: true });
        }
        // case 2: already loading this track — just update _pendingSeekTime
        // the in-flight canplay handler will pick it up

    } else {
        // case 3: same track, not loading — direct seek
        this.audio.currentTime = playback_time;
        if (is_playing) this._radioPlay();
        if (!is_playing && !this.audio.paused) this.audio.pause();
    }
}

the syncTimer debounce (80ms) collapses rapid Sync messages — common during a seek — into a single call to syncToBroadcaster. without this, a fast seek could queue up dozens of competing load/seek operations.

AutoNext — finishing your own copy before switching

when a broadcaster's track ends and they move to the next, they send AutoNext. listeners do not immediately jump — they have a few seconds of buffered audio left from the current track that they should play out. pendingAutoNextIndex queues the next track index, and a one-shot ended listener triggers the switch when the listener's local copy actually finishes:

// on receiving AutoNext:
this.pendingAutoNextIndex = msg.next_media_index;

this.audio.addEventListener("ended", () => {
    if (this.pendingAutoNextIndex === null) return;
    const idx = this.pendingAutoNextIndex;
    this.pendingAutoNextIndex = null;
    this.audio.src = `/stream/id/${mediaId}`;
    this.audio.load();
    this.audio.addEventListener("canplay", () => {
        this.audio.currentTime = seekTo;
        this._radioPlay();
    }, { once: true });
}, { once: true });

some Bluetooth codecs re-negotiate when audio is interrupted mid-stream. the re-negotiation takes 200–500ms and produces a harsh click. muting during seeks and unmuting 1 second after playing fires covers the codec setup window:

const btUnmuteOnPlaying = () => {

};
playlist.js javascript

PlaylistManager handles loading, rendering, filtering, drag-and-drop reordering, search, and downloads. it maintains two parallel arrays — the user's custom order and the server's original order — and translates between them for broadcasting.

originalMedias vs medias — why two arrays?

this.originalMedias = serverMedias;  // server order, never changes
this.medias         = [...serverMedias];  // user's display order, can be reordered

the server's /stream/{index} endpoint is index-based. when broadcasting, the app must report the server's index for the currently playing file. if the user reorders their playlist, this.medias[3] might be a different file than this.originalMedias[3]. getServerIndexById() translates from media ID to server index by searching originalMedias:

getServerIndexById(mediaId) {
    return this.originalMedias.findIndex((s) => s.id === mediaId);
}

custom order persistence

saveCustomOrder() {
    const orderIds = this.medias.map((s) => s.id);
    localStorage.setItem(this.storageKey, JSON.stringify(orderIds));
}

loadCustomOrder() {
    const savedOrder = localStorage.getItem(this.storageKey);
    const orderIds = JSON.parse(savedOrder);
    // filter out IDs that no longer exist on the server (deleted files)
    const validIds = orderIds.filter((id) => this.medias.some((s) => s.id === id));
    // build ordered array, append any new server files at the end
    const ordered = validIds.map((id) => this.medias.find((s) => s.id === id));
    this.medias.forEach((m) => { if (!validIds.includes(m.id)) ordered.push(m); });
    this.medias = ordered;
}

the storage key includes the session ID (playlist_order_${sessionId}), so each browser tab gets its own independent ordering. the validation step against the current server list means deleted files disappear cleanly and newly uploaded files appear at the end.

the filter system

the filter (All / Audio / Video / Other) gates only next/prev navigation. direct play clicks always work regardless of the active filter:

_filteredMedias() {
    if (this.mediaFilter === "all") return this.medias;
    return this.medias.filter((m) =>
        this.mediaFilter === "video" ? this.isVideo(m) : !this.isVideo(m)
    );
}

getNextMedia() {
    const pool = this._filteredMedias();
    const currentInPool = pool.findIndex((m) => m.id === this.currentMediaId);
    return pool[(currentInPool + 1) % pool.length];
}

touch drag-and-drop on mobile

mobile browsers fire touchmove on the viewport, not on DOM elements. the handler must listen globally and use document.elementFromPoint(x, y) to find what is under the finger. the event is attached to the drag handle only (not the whole row) to preserve normal scrolling.

handleTouchMove(e) {
    e.preventDefault();  // prevents scroll while dragging
    const touch = e.touches[0];

    // move the clone to follow the finger
    this.touchClone.style.top = `${touch.clientY - this.touchGrabOffsetY}px`;

    // auto-scroll the list when clone reaches the edge
    const container = document.getElementById("playlist-display");
    if (cloneRect.bottom > cRect.bottom + 30) container.scrollTop += 8;

    // find what the finger is over
    this.touchClone.style.display = "none";  // hide clone so it doesn't block elementFromPoint
    const target = document.elementFromPoint(touch.clientX, touch.clientY);
    this.touchClone.style.display = "";

    const targetItem = target?.closest(".playlist-item");
    const targetIndex = parseInt(targetItem?.dataset.index);

    // only reorder once the clone's edge crosses the target's midpoint
    const targetMid = targetRect.top + targetRect.height / 2;
    const movingDown = targetIndex > this.touchDragCurrentIndex;
    if (movingDown && touch.clientY < targetMid) return;
    if (!movingDown && touch.clientY > targetMid) return;

    // do the swap
    const media = this.medias[this.touchDragCurrentIndex];
    this.medias.splice(this.touchDragCurrentIndex, 1);
    this.medias.splice(targetIndex, 0, media);
    navigator.vibrate(15);  // haptic feedback
    this.renderForDrag(targetIndex);
}

main.js javascript

bootstrap file. sets up the dual audio/video element system, patches play() on both elements to handle seek races, wires all button listeners, registers the OS media session, and kicks off the WebSocket connection and playlist load.

the dual media element system

the page always has both <audio> and <video> elements in the DOM. only one is visible at a time. window._activeMedia always points to whichever is active. switchMediaElement(toVideo) swaps them:

window.switchMediaElement = function(toVideo) {
    const incoming = toVideo ? videoEl : audioEl;
    const outgoing = toVideo ? audioEl : videoEl;

    if (incoming === window._activeMedia) return;

    outgoing.pause();
    outgoing.src = "";
    outgoing.classList.add("hidden");

    seekPending = false;  // reset to prevent deferred play on wrong element

    incoming.classList.remove("hidden");
    window._activeMedia = incoming;

    if (window.player) window.player.audio = incoming;  // update RadioPlayer's reference
};

the initial active element is set server-side: if the current track is a video, the server renders the audio element with class="hidden" and omits its <source> tag. this prevents the browser from ever trying to decode a video file through the audio element before JavaScript runs.

the seek-aware play() patch

there is a race condition in the HTML5 media API: if you set src, call load(), set currentTime, and immediately call play(), the play often fails with an AbortError because the seek is still pending. the fix is to defer play() until the seeked event fires:

function patchPlay(el, originalPlay) {
    el.play = () => {
        if (!seekPending) return originalPlay();  // normal path
        return new Promise((resolve, reject) => {
            el.addEventListener(
                "seeked",
                () => originalPlay().then(resolve).catch(reject),
                { once: true }
            );
        });
    };
}
patchPlay(audioEl, originalPlayAudio);
patchPlay(videoEl, originalPlayVideo);

both elements are patched because switching media types calls play() on whichever element just became active, and it might still be seeking.

upload.js javascript

handles file uploads via XHR with a real-time progress bar. batch uploads report per-file successes and failures without reloading the page.

why XHR instead of fetch?

the Fetch API does not expose upload progress events in a straightforward way. XMLHttpRequest has xhr.upload.addEventListener("progress", ...) which fires repeatedly during the upload with event.loaded and event.total. this drives the progress bar.

    if (e.lengthComputable)
        bar.style.width = (e.loaded / e.total) * 100 + "%";

Latency compensation in detail

keeping listeners in sync with the broadcaster requires compensating for two distinct network legs.

the two latency legs
leg 1 — broadcaster to server: the broadcaster stamps Date.now() on every outgoing message. the server subtracts it from its own wall clock on receipt. this is an approximation — it includes both direction latencies — but since the jitter is symmetrical it works well as a rough estimate.

leg 2 — server to listener: the browser records Date.now() as _tuneInSentAt when sending TuneIn. when the first Sync arrives, the elapsed time since _tuneInSentAt is added to the received playback_time. this covers how long the Sync spent travelling from server to listener.

on ongoing BroadcastUpdates, leg 1 is applied server-side before the Sync is sent. leg 2 was already applied during TuneIn and the listener's audio element has been playing since, so no additional adjustment is needed — drift is small over short periods and the heartbeat keeps positions fresh.

on TuneIn, leg 1 is baked into the adjusted position the server sends. leg 2 is applied by the browser on the first Sync. together they aim the listener at approximately where the broadcaster is right now, not where they were when the server last received an update.

04 — retrospective

Design Decisions

decisionrationale
no database sessions and broadcast state are ephemeral. the only persistent state is the playlist (a folder of files) and user preferences (localStorage). adding SQLite would introduce migrations, a dependency, and write-ahead-log maintenance for no gain.
no frontend framework the UI has two interactive classes and six auxiliary files. a framework's abstraction overhead would exceed the complexity it manages. vanilla JS with clear ownership is easier to reason about and debug.
DashMap everywhere all high-frequency paths (heartbeat, TuneIn, BroadcastUpdate) hit different session keys. sharded concurrency eliminates the global write bottleneck of a single RwLock<HashMap>.
PreparedMessage + Arc serialising once and cloning a pointer is O(1) per additional listener. serialising per-listener would be O(n). at scale the difference is significant.
always 206, never 200 browsers treat a 200 response as a non-seekable stream on some codecs and delay playback by downloading the moov atom first. 206 signals "range-aware" universally.
video conversion at upload time converting on every stream request would block CPUs and add first-byte latency. converting once at upload gives instant subsequent streaming. ffmpeg's faststart flag ensures the moov atom is at the front for fast playback start.
DeleteOnDrop for temp files error handling in async code is subtle — a return early from a handler due to a client disconnect would skip any cleanup at the end of the function. RAII binds cleanup to object lifetime, which Rust guarantees regardless of how the function exits.
biased select! in the send task Tokio's default random branch selection prevents starvation but adds unnecessary indirection on the hottest path. the broadcaster's sync channel is always the highest-priority source; checking it first reduces end-to-end latency on the critical seek path.
admin panel is public designed for a trusted home network. session IDs are non-guessable UUIDs. for a multi-tenant deployment this would need authentication.
TCP_NODELAY on every socket Nagle's algorithm batches small writes for bulk efficiency. WebSocket sync frames are tiny and latency-sensitive. disabling batching costs negligible bandwidth and noticeably reduces seek-to-sync delay.
